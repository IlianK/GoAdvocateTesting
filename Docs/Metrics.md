# Metrics
The comparable metrics are extracted from the restructured ADVOCATE results under `results/…/run-*/` and written into CSVs generated by `advocate-runner compare …`.

To keep this flexible and maintainable, metric extraction is driven by two YAML files. A user-editable `metrics_select.yaml` (user-editable), containing the metrics the user wants in the comparison CSVs and `metrics_def.yaml` used for maintenance and compatibility with ADVOCATE.

---

## File Overview

### [`metrics_select.yaml`](../metrics_select.yaml)
This file controls which columns appear in the comparison CSVs and in which order.

It has two main sections: `default` and `specific`, and a distinction between `sources` and `derived`.

#### Use cases
- Use `default` for your typical “comparison table”.
- Use `specific` to quickly compare only a small subset of metrics (e.g., only bug counts). If non-empty, it overrides default entirely.

#### Sources and Derived
- `sources`: lists direct CSV columns (read from result CSV headers).
- `derived`: lists derived metrics (computed via rules in `metrics_def.yaml`).

The order in the YAML is respected:
- `sources` are emitted in a stable source-order (`statsAnalysis` -> `statsFuzz` -> `statsAll` -> `statsTrace` -> `timesTotal`/`timesDetail`)
- `derived` are emitted exactly in their listed order

---

### [`metrics_def.yaml`](../internal/metrics/metrics_def.yaml)
This file defines directory names used inside a run folder, the filename patterns for each source and the wiring needed to derive metrics/ how to compute them. 
Used for compatibility if ADVOCATE changes filenames/columns over time.


#### Extraction workflow
For each discovered run (`discovery.Run`), the extractor:

1) Loads selection + definitions
- loads `internal/metrics/metrics_def.yaml`
- loads `metrics_select.yaml`
- chooses either:
    - specific (if non-empty)
    - else default

2) Extracts direct metrics (`sources`)
For each selected `source` (e.g. `statsAnalysis`, `statsAll`, …):
- locate the matching file pattern from metrics_def.yaml
- parse the CSV header + first non-empty data row
- for each requested column in `metrics_select.yaml`, read the value by header name

3) Computes derived metrics (`derived`)
Each derived metric is computed using its rule in `metrics_def.yaml`.
Derived metrics **are evaluated in the exact order** listed in `metrics_select.yaml`, which is important when **one derived metric depends on another** (e.g. `Unique_Bugs` depends on `Bug_Types`).


#### Metric Sources
A run folder typically contains:
```
output/
- output.log
- results_machine.log
- results_readable.log

stats/
- statsAll_*
- statsAnalysis_*
- statsFuzz_* 
- statsTrace_*

times/ 
- times_total_*.csv
- times_detail_*.csv

bugs/
- bug_*.md 
```

The extractor uses the dirs and files entries in `metrics_def.yaml` to find the `stats*_csv` files reliably.


#### Derived Metrics

##### Bug_Types
**Meaning:** Which bug codes were observed for this run (e.g., `P06`, `A07`, `L02`) as a semicolon-separated list.  
**Derived from:** `stats/statsAll_*`, by scanning the `statsAll` CSV row for the “unique detected” bug-code columns (e.g., `NoUnique...<CODE>`). Any code with a value `> 0` is included.


##### Unique_Bugs
**Meaning:** Count of distinct bug codes found (i.e., the number of entries in `Bug_Types`).  
**Derived from:** `stats/statsAll_*`, by scanning the `statsAll` CSV row for the “unique detected” bug-code columns (e.g., `NoUnique...<CODE>`) and building the sum over all.


##### Total_Bugs
**Meaning:** Number of bug report files produced for the run (how many reports were written).  
**Derived from:** bug report files on disk (`bugs/bug_*.md` and/or `bug_*.md` depending on layout), by counting files matching the configured `bugReports` pattern in the `bugs/` subfolder.


##### Total_Runs
**Meaning:** Total number of fuzzing executions/mutations for the run.  
**Derived from:** `stats/statsFuzz_*` as Alias of `NrMut` 


##### Replays_Written
**Meaning:** Total number of replay attempts that were generated/written (across all bug types).  
**Derived from:** `stats/statsAll_*`, by summing all `statsAll` columns representing “replay written” counts across bug codes.


##### Replays_Successful
**Meaning:** Total number of replay attempts that succeeded (across all bug types).  
**Derived from:** `stats/statsAll_*`, by summing all `statsAll` columns representing “replay successful” counts across bug codes.


##### Confirmed_Replays
**Meaning:** How many issues were confirmed/resolved via replay (e.g., panics verified via replay + leaks resolved via replay).  
**Derived from:** `stats/statsAnalysis_*`, by building the sum of the two analysis CSV columns:
- `NrPanicsVerifiedViaReplayTotal`
- `NrLeaksResolvedViaReplayTotal`


##### Total_Time_s
**Meaning:** Total wall-clock duration of the run (seconds).  
**Derived from:** `times_total_*.csv` (either in run root or under `times/`), by reading the row where `TestName == "Total"` and extract the `Time` column.


##### Rec_s
**Meaning:** Time spent in recording (seconds).  
**Derived from:** `times_detail_*.csv`, by extracting the `Recording` column.


##### Ana_s
**Meaning:** Time spent in analysis (seconds).  
**Derived from:** `times_detail_*.csv`, by extracting the `Analysis` column.


##### Rep_s
**Meaning:** Time spent in replay (seconds).  
**Derived from:** `times_detail_*.csv`, by extracting the `Replay` column.

## Notes
- Direct columns must exist in the CSV header to be extracted (exact name match)
- Derived metrics depend on correct wiring in `metrics_def.yaml`
- If ADVOCATE changes a filename/metric name, you usually only need to update `metrics_def.yaml`, not the code `./internal/metrics`
- If `specific` is used, it **replaces** `default` entirely. **It does not merge**
